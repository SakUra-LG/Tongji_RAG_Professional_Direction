# SynapseQ 设计与实现审计

## 已实现的设计主线

- 前端使用统一自动问答入口，不允许用户自行选择数据模块。
- 访客只检索公开数据；学生和教师自动连接公开、校内爬取及本人个人数据。
- 自动流水线按“个人结构化数据 -> 个人向量 -> FAQ/爬取数据 -> 通用大模型”
  的顺序回答。
- 个人结构化查询在问题重写之前执行，绩点和课表不依赖外部大模型可用性。
- `ACADEMIC_URLS` 的爬取结果保存在 `rag_knowledge`，统一入口将其视为校内数据。
- 51 条人工 FAQ 在外部模型调用前进行本地匹配，并按标准问题去重同步到 Milvus。
- DashScope 不可用时，Milvus 检索降级为本地关键词匹配；数据库命中后可直接返回
  资料摘要，避免已有数据也完全不可用。
- 内部通知使用结构化通知的 `dept_id + audience` 双重过滤。
- 个人检索使用 JWT 用户 ID 读取当前用户档案，并继续兼容按 `user_id` 过滤的个人向量。
- Redis 保存会话和最近消息，支持问题重写。
- Milvus 向量分数与 Jieba 关键词覆盖率进行加权重排。
- FastAPI 通过 SSE 流式返回回答。
- MySQL 保存用户、师生档案、课表、成绩、通知和爬取任务元数据。

## 本次已修复

1. `.env.example` 中包含真实 DashScope Key。
2. JWT 密钥写死在代码中，环境变量配置不生效。
3. CORS 使用通配符并同时允许凭据。
4. 刷新令牌原文被直接放入 Redis Key。
5. 会话历史接口未验证会话是否属于当前用户。
6. 前端将用户密码明文保存到 `localStorage`。
7. LLM 输出直接进入 `v-html`，存在 XSS 风险。
8. 前端 API 写死为 `localhost:8000`，远程部署不可用。
9. Docker 初始化脚本每次启动都会删除全部 Milvus 集合。
10. Docker 服务缺少完整健康检查和自动初始化顺序。
11. Milvus 数量脚本与当前 `pymilvus` 版本不兼容。
12. 底层模型服务异常会原样返回到前端，泄露内部网络和调用栈信息。
13. 学生和教师缺少结构化个人档案、课表、成绩数据模型及独立查询接口。
14. 内部通知只按院系过滤，学生可能检索到同院系中明确面向教职工的通知。
15. 访问学者角色、王学者演示账号和前端残留分支与当前需求冲突。
16. 前端允许用户手动选择四类模块，导致师生可能在公开模块中无法命中自己的绩点或课表。
17. 个人结构化查询位于 LLM 查询改写之后，外部模型异常时会影响本地个人数据查询。

## 仍需后续完善

### 高优先级

- 真实统一身份认证尚未接入，当前仍是课程演示账户。
- 回答来源尚未作为结构化 SSE 数据返回，前端也未展示真实引用卡片。
- HTTPS 需要由部署环境的反向代理或网关提供。
- Milvus 中的历史内部和个人文本缺少完整可恢复 CSV 备份；新增结构化数据已保存到 MySQL。
- 生成内容缺少独立的内容安全过滤与审计流水。

### 中优先级

- `App.vue` 仍偏大，后续应继续拆分登录、会话和右侧信息面板组件。
- RAG 阈值、Top-K 和重排权重写死在代码中，缺少离线评测与配置化。
- 知识库更新尚无版本号、双集合切换和自动回滚机制。
- 日志会打印检索文本预览，生产环境需要脱敏并降低日志级别。
- Redis 会话采用固定 7 天 TTL，与需求文档中的“会话结束清除”仍有差异。
- 目前没有自动化测试覆盖权限矩阵、会话越权、检索过滤和 SSE。
